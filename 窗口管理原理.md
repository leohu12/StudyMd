## 一、窗口是什么

窗口就是一个**独立的画板**：

| 窗口类型 | 示例 |
|----------|------|
| 应用窗口 | Activity 的界面 |
| 子窗口   | 对话框、PopupWindow |
| 系统窗口 | 状态栏、导航栏、输入法、Toast、锁屏 |

### 1.1 核心角色

```text
┌─────────────────────────────────────────────────────────┐
│ Activity                                               │
│ （剧本：创建窗口、贴布局）                             │
└─────────────────────────┬───────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│ WindowManager                                          │
│ （舞台助理：向 WMS 申请窗口）                          │
└─────────────────────────┬───────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│ WindowManagerService (WMS)                              │
│ （舞台总指挥：管理窗口顺序、位置）                     │
└─────────────────────────┬───────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│ SurfaceFlinger                                          │
│ （灯光合成师：合成图像到屏幕）                         │
└─────────────────────────────────────────────────────────┘
```

### 1.2 Activity 显示窗口的流程

```text
创建画板
└── Activity 启动时，系统创建 PhoneWindow

贴布局
└── setContentView() 把布局设置到 DecorView

申请上台
└── WindowManager.addView() 通知 WMS

安排层级
└── WMS 为窗口分配 z-order（层级）

合成显示
└── SurfaceFlinger 混合所有窗口图像，送显
```

## 二、窗口层级（z-order）

系统用**类型**来决定层级，数字越大越在上面：

| 类型     | 层级范围     | 示例                   |
|----------|--------------|------------------------|
| 应用窗口 | 1～99        | Activity               |
| 子窗口   | 1000～1999   | 对话框、PopupWindow    |
| 系统窗口 | 2000～2999   | 状态栏、输入法、Toast、锁屏 |

### 2.1 叠放顺序

```text
┌─────────────────────────────────────────────┐
│ 锁屏（最高层）                              │
├─────────────────────────────────────────────┤
│ 状态栏 / 导航栏                             │
├─────────────────────────────────────────────┤
│ 输入法（弹出时）                            │
├─────────────────────────────────────────────┤
│ 系统悬浮窗                                  │
├─────────────────────────────────────────────┤
│ 对话框                                      │
├─────────────────────────────────────────────┤
│ Activity（当前应用）                        │
├─────────────────────────────────────────────┤
│ 桌面（最底层）                              │
└─────────────────────────────────────────────┘
```

### 2.2 为什么需要这种设计？

- **安全性**：系统窗口（如输入法）可以盖在应用上面，但应用无法盖住系统窗口
- **用户体验**：对话框、Toast 可以随时弹出，Activity 可以随时切换
- **隔离性**：每个窗口独立绘制，一个窗口卡顿不会影响其他窗口

## 三、触摸事件分发

### 3.1 事件查找流程

```text
当你点击屏幕时，系统按以下步骤找到目标窗口：
触摸事件发生（坐标 x=500, y=800）

WMS 从上到下遍历窗口
└── 状态栏：点在我区域吗？→ 否
└── 输入法：点在我区域吗？→ 否
└── 对话框：点在我区域吗？→ 是
└── 对话框愿意接收吗？→ 是
└── 事件交给对话框

对话框内部的 View 继续分发事件
```

### 3.2 事件分发规则

```text
┌─────────────────────────────────────────────────┐
│ 最上层窗口                                      │
│ （优先接收触摸事件）                            │
├─────────────────────────────────────────────────┤
│ 第二层窗口                                      │
│ （只有上层不处理时才收到事件）                  │
├─────────────────────────────────────────────────┤
│ 底层窗口                                        │
│ （很少能收到事件）                              │
└─────────────────────────────────────────────────┘

规则：自上而下，上层优先，拦截即止
```

### 3.3 焦点管理

- **键盘输入**：只能有一个窗口获得焦点
- **焦点转移**：通常给最上层的可交互窗口
- **输入法窗口**：不抢焦点，但会让当前 Activity 调整布局

## 四、多窗口竞合处理

### 4.1 竞合场景

当多个窗口同时存在时，可能发生：

| 场景         | 竞合问题          | 系统处理             |
|--------------|-------------------|----------------------|
| 多个悬浮窗   | 谁在上谁在下      | 后添加的默认在上     |
| 输入法弹出   | 挡住输入框        | 通知 Activity 调整布局 |
| Toast 排队   | 多个 Toast 同时出现 | 排队显示，不叠加  |
| 系统警告     | 覆盖当前应用      | 强制提升到顶层       |

### 4.2 核心解决机制

```text
多窗口竞合
│
├── 固定层级表 —— 按类型决定谁在上面
├── 自上而下事件分发 —— 最上层优先处理触摸
└── 单一焦点 —— 只有一个窗口能接收键盘
```

### 4.3 开发者限制

普通应用**无权改变窗口层级**，这是 Android 的安全设计：

- 防止恶意应用覆盖其他应用或系统界面
- 悬浮窗需要申请 `SYSTEM_ALERT_WINDOW` 权限
- 使用 `TYPE_APPLICATION_OVERLAY` 类型
- 无法盖住状态栏、输入法或锁屏

## 五、车载系统的窗口管理

### 5.1 车载 vs 手机的差异

| 维度   | 手机         | 车载                   |
|--------|--------------|------------------------|
| 屏幕数量 | 单屏         | 多屏（中控、仪表、后排） |
| 安全要求 | 低           | 高（倒车影像、警告必须显示） |
| 用户区域 | 单一用户     | 多用户（司机、副驾、后排） |
| 窗口隔离 | 无           | 按音区隔离             |

### 5.2 多屏独立竞合

车载系统通常有多个屏幕，每个屏幕有自己的窗口栈：

```text
┌────────────┐ ┌────────────┐ ┌────────────┐
│ 中控屏     │ │ 仪表盘     │ │ 后排娱乐   │
├────────────┤ ├────────────┤ ├────────────┤
│ 导航 App   │ │ 车速显示   │ │ 视频 App   │
│ 音乐 App   │ │ 警告灯     │ │ 游戏 App   │
│ 空调控制   │ │ 导航箭头   │ │ 音乐 App   │
└────────────┘ └────────────┘ └────────────┘
独立栈    独立栈    独立栈
```

### 5.3 安全关键窗口最高优先级

某些窗口具有**最高优先级**，甚至可以覆盖系统 UI：

| 窗口类型   | 优先级 | 说明                           |
|------------|--------|--------------------------------|
| 倒车影像   | 🔴 最高 | 挂倒挡立即显示，覆盖一切       |
| 碰撞预警   | 🔴 最高 | 系统级警告，强制弹出           |
| 故障提示   | 🟡 高   | 油量低、胎压异常               |
| 导航提示   | 🟢 中   | 转弯提示、限速提醒             |
| 媒体播放   | ⚪ 低   | 音乐、视频                     |

### 5.4 多用户场景

现代车载系统支持为不同座位分配独立窗口区域：

```text
┌──────────────────────────────┐
│ 中控屏                       │
├────────────┬───────────────┤
│ 司机区     │ 乘客区         │
│ 导航 App   │ 视频 App       │
│ 通话界面   │ 游戏 App       │
│ 空调快捷操作 │ 音乐控制     │
└────────────┴───────────────┘
```

- 每个区域有自己的窗口栈
- 系统窗口（如空调调节）可能同时影响两个区域
- 焦点可以在区域间切换

### 5.5 窗口焦点与输入焦点分离

在车载系统中，**窗口焦点**和**输入焦点**可能分离：

- **窗口焦点**：哪个窗口在最上面
- **输入焦点**：哪个窗口接收触摸/按键

例如：中控屏上导航窗口在顶层，但用户正在用方向盘按键控制音乐窗口的播放/暂停。这时输入焦点在音乐窗口，而显示焦点在导航窗口。

## 📖 总结

| 概念           | 职责             | 关键点                  |
|----------------|------------------|-------------------------|
| **Window**     | 独立的画板       | 每个窗口有自己的 Surface |
| **WMS**        | 窗口管理服务     | 管理 z-order、位置、大小 |
| **SurfaceFlinger** | 图像合成     | 混合所有窗口图像         |
| **z-order**    | 窗口层级         | 按类型分配，数字越大越靠上 |
| **触摸分发**   | 事件查找         | 从上到下，上层优先       |

**一句话总结**：窗口管理是系统安排多个独立画板的层次、位置和事件传递，让它们有序地组合成你看到的屏幕画面。
