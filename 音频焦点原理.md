## 一、麦克风栈模型

整个系统只有**一支麦克风**，谁拿到谁说话。

### 1.1 栈结构
┌─────────────────────────────────────┐
│ 栈顶（当前拥有焦点） │
│ 音乐 App（正在播放） │
├─────────────────────────────────────┤
│ 栈中（等待恢复） │
│ 导航 App（被暂停） │
├─────────────────────────────────────┤
│ 栈底（长期无焦点） │
│ 游戏 App（已停止播放） │
└─────────────────────────────────────┘

### 1.2 栈操作规则

| 操作 | 说明 | 效果 |
|------|------|------|
| **压栈** | 新应用申请焦点 | 新应用到栈顶，原栈顶下沉 |
| **弹栈** | 应用释放焦点 | 栈顶移除，下一个应用自动获得焦点 |
| **清栈** | 系统强制移除 | 所有应用失去焦点 |

### 1.3 核心组件
┌─────────────────────────────────────────────────────────┐
│ App 层 │
│ MediaPlayer.start() / AudioTrack │
└─────────────────────────┬───────────────────────────────┘
│ requestAudioFocus()
▼
┌─────────────────────────────────────────────────────────┐
│ AudioManager（焦点管理器） │
│ 维护麦克风栈，通知焦点变化 │
└─────────────────────────┬───────────────────────────────┘
│
▼
┌─────────────────────────────────────────────────────────┐
│ AudioService（系统服务） │
│ 处理焦点请求，分发通知 │
└─────────────────────────────────────────────────────────┘

## 二、四种申请方式

### 2.1 类型对比

| 申请类型 | 大白话 | 对其他App的通知 | 结束后 | 典型场景 |
|----------|--------|-----------------|--------|----------|
| `GAIN` | 我要开演唱会 | 永久失去 (LOSS) | 不自动恢复 | 音乐App播放歌曲 |
| `GAIN_TRANSIENT` | 我插句话 | 暂时失去 (LOSS_TRANSIENT) | 自动恢复 | 导航播报、微信语音 |
| `GAIN_TRANSIENT_MAY_DUCK` | 我小声嘀咕 | 闪避 (CAN_DUCK) | 自动恢复音量 | 短信提示音 |
| `GAIN_TRANSIENT_EXCLUSIVE` | 我要清场 | 暂时失去且独占 | 自动恢复 | 语音助手、录音 |

### 2.2 详细说明

#### 🎤 GAIN（开演唱会）

```java
AudioFocusRequest request = new AudioFocusRequest.Builder(AudioManager.AUDIOFOCUS_GAIN)
    .setOnAudioFocusChangeListener(listener)
    .build();
audioManager.requestAudioFocus(request);
意图：长期占用麦克风

对其他App：通知永久失去焦点，建议彻底停止播放

结束后：不会自动恢复其他App

🗣️ GAIN_TRANSIENT（插句话）
java
AudioFocusRequest request = new AudioFocusRequest.Builder(AudioManager.AUDIOFOCUS_GAIN_TRANSIENT)
    .setOnAudioFocusChangeListener(listener)
    .build();
意图：临时打断，说完就走

对其他App：通知暂时失去焦点，建议暂停播放

结束后：自动恢复其他App

🔊 GAIN_TRANSIENT_MAY_DUCK（小声嘀咕）
java
AudioFocusRequest request = new AudioFocusRequest.Builder(AudioManager.AUDIOFOCUS_GAIN_TRANSIENT_MAY_DUCK)
    .setOnAudioFocusChangeListener(listener)
    .build();
意图：短暂发声，其他App可以继续

对其他App：通知闪避，建议降低音量

结束后：自动恢复其他App的音量

🔇 GAIN_TRANSIENT_EXCLUSIVE（清场）
java
AudioFocusRequest request = new AudioFocusRequest.Builder(AudioManager.AUDIOFOCUS_GAIN_TRANSIENT_EXCLUSIVE)
    .setOnAudioFocusChangeListener(listener)
    .build();
意图：需要安静环境（录音、语音识别）

对其他App：通知暂时失去焦点，建议完全静音

结束后：自动恢复其他App

2.3 焦点变化监听
java
private AudioManager.OnAudioFocusChangeListener listener = new AudioManager.OnAudioFocusChangeListener() {
    @Override
    public void onAudioFocusChange(int focusChange) {
        switch (focusChange) {
            case AudioManager.AUDIOFOCUS_GAIN:
                // 重新获得焦点：恢复播放
                resumePlayback();
                break;
            case AudioManager.AUDIOFOCUS_LOSS:
                // 永久失去焦点：停止播放，释放资源
                stopPlayback();
                break;
            case AudioManager.AUDIOFOCUS_LOSS_TRANSIENT:
                // 暂时失去焦点：暂停播放，保持进度
                pausePlayback();
                break;
            case AudioManager.AUDIOFOCUS_LOSS_TRANSIENT_CAN_DUCK:
                // 闪避：降低音量到20%
                lowerVolume(0.2f);
                break;
        }
    }
};
三、音频策略（AudioPolicy）
3.1 为什么需要策略？
如果只有麦克风栈，只能解决“谁说话”的问题，但实际场景更复杂：

❓ 导航播报应该从哪个喇叭出声？

❓ 音乐和导航同时播，音量怎么配合？

❓ 蓝牙耳机连接后，所有声音都走耳机吗？

❓ 后排看电影，前排导航，声音怎么隔离？

音频策略就是解决这些问题的“总导演”。

3.2 核心架构
text
┌─────────────────────────────────────────────────────────┐
│                    App 层                               │
│      指定 audio attributes（USAGE_MEDIA 等）             │
└─────────────────────────┬───────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│          AudioPolicyService（策略制定者）                │
│   决定：声音发到哪个设备、怎么混音、谁优先                 │
│   输入：音频属性(usage)、当前设备状态、策略配置文件        │
│   输出：输出设备(扬声器/耳机/蓝牙)、音量曲线              │
└─────────────────────────┬───────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│            AudioFlinger（策略执行者）                    │
│   创建 PlaybackThread、混音、写入 HAL                    │
└─────────────────────────┬───────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────┐
│              Audio HAL（硬件抽象层）                     │
│              厂商实现的 so 库                            │
└─────────────────────────────────────────────────────────┘
3.3 策略配置文件
车载音频策略通过 XML 配置文件定义：

audio_policy_engine_configuration.xml

xml
<!-- 定义产品策略：将音频用途映射到处理策略 -->
<productStrategies>
    <!-- 媒体策略 -->
    <strategy name="STRATEGY_MEDIA">
        <attributes usage="USAGE_MEDIA"/>
        <attributes usage="USAGE_GAME"/>
    </strategy>
    
    <!-- 导航策略：优先级更高 -->
    <strategy name="STRATEGY_NAVIGATION">
        <attributes usage="USAGE_ASSISTANCE_NAVIGATION_GUIDANCE"/>
    </strategy>
    
    <!-- 通话策略 -->
    <strategy name="STRATEGY_PHONE">
        <attributes usage="USAGE_VOICE_COMMUNICATION"/>
    </strategy>
</productStrategies>

<!-- 定义音量组 -->
<volumeGroups>
    <group name="media" stream="AUDIO_STREAM_MUSIC">
        <deviceCategory name="DEVICE_CATEGORY_SPEAKER"/>
        <deviceCategory name="DEVICE_CATEGORY_HEADPHONES"/>
    </group>
    
    <group name="navigation" stream="AUDIO_STREAM_MUSIC">
        <deviceCategory name="DEVICE_CATEGORY_SPEAKER"/>
    </group>
</volumeGroups>
四、从建议到强制
4.1 焦点的“建议”本质
音频焦点是建议机制，不是强制性的：

java
// App 收到焦点变化通知后，可以“不听话”
listener.onAudioFocusChange(LOSS) {
    // 我偏要继续播放
    continuePlayback();  // ❌ 系统不会阻止
}
4.2 强制策略的实现
车载系统可以通过策略注入实现强制控制：

java
// 系统 App 注入自定义策略
AudioPolicy policy = new AudioPolicy.Builder(context)
    .setIsAudioPolicy(true)
    .addMix(mix)  // 定义混音规则
    .build();

audioManager.registerAudioPolicy(policy);
策略注入后的强制效果：

场景	普通焦点	策略注入后
导航播报时	音乐App可以选择不闪避	系统强制压低音乐音量
语音通话时	音乐App可以继续播放	系统强制静音音乐
倒车雷达报警	其他声音不受影响	系统强制所有声音降低
4.3 权限要求
策略注入需要系统权限：

xml
<uses-permission android:name="android.permission.MODIFY_AUDIO_ROUTING" />
普通第三方 App 无法获得此权限，只有系统预装 App 才能实现。

五、车载多音区管理
5.1 多音区概念
现代车载系统将车内分为多个独立音区：

text
┌─────────────────────────────────────────────────┐
│                   驾驶区                         │
│          导航播报、通话、雷达报警                  │
├─────────────────────────────────────────────────┤
│                   副驾区                         │
│          视频、音乐、游戏                         │
├─────────────────────────────────────────────────┤
│                   后排左区                        │
│          电影、游戏、独立音乐                      │
├─────────────────────────────────────────────────┤
│                   后排右区                        │
│          电影、游戏、独立音乐                      │
└─────────────────────────────────────────────────┘
5.2 多音区配置
car_audio_configuration.xml

xml
<audioZones>
    <!-- 驾驶区：主音区 -->
    <audioZone name="driver" isPrimary="true">
        <device address="bus0_media" volumeGroup="media"/>
        <device address="bus0_nav" volumeGroup="navigation"/>
        <device address="bus0_phone" volumeGroup="phone"/>
    </audioZone>
    
    <!-- 副驾区 -->
    <audioZone name="passenger">
        <device address="bus1_media" volumeGroup="media_passenger"/>
    </audioZone>
    
    <!-- 后排左区 -->
    <audioZone name="rear_left">
        <device address="bus2_media" volumeGroup="media_rear"/>
    </audioZone>
    
    <!-- 后排右区 -->
    <audioZone name="rear_right">
        <device address="bus3_media" volumeGroup="media_rear"/>
    </audioZone>
</audioZones>
5.3 音区隔离效果
text
驾驶区（导航播报）
    │
    ├── 导航语音 → 驾驶区扬声器（独立）
    ├── 音乐 → 驾驶区扬声器（可独立调节）
    └── 通话 → 驾驶区扬声器（私密）

副驾区（看视频）
    │
    ├── 视频声音 → 副驾区扬声器/耳机
    └── 与驾驶区完全隔离

后排区（玩游戏）
    │
    ├── 游戏声音 → 后排区扬声器/耳机
    └── 左右独立，互不干扰
5.4 动态策略注入
车载系统可以动态调整策略：

java
// 注册动态策略混音规则
AudioMix mix = new AudioMix.Builder()
    .setFormat(bitstreamFormat)
    .setRouteFlags(routeFlags)
    .addRule(AudioMix.RULE_MATCH_ATTRIBUTE_USAGE, USAGE_MEDIA)
    .build();

AudioPolicy policy = new AudioPolicy.Builder(context)
    .addMix(mix)
    .build();

audioManager.registerAudioPolicy(policy);
匹配规则：

规则	含义	应用场景
RULE_MATCH_ATTRIBUTE_USAGE	按音频用途匹配	导航语音只走驾驶区
RULE_MATCH_UID	按 App UID 匹配	特定音乐 App 走后排
RULE_MATCH_USERID	按用户 ID 匹配	乘客投屏内容走对应音区
📖 总结
概念	职责	关键点
音频焦点	谁拿到麦克风	栈结构，四种申请类型
AudioPolicy	声音往哪走	配置文件定义，策略注入
多音区	分区独立管理	驾驶区、副驾区、后排区
策略注入	强制行为	需要系统权限
一句话总结：音频焦点解决“谁说话”的问题，音频策略解决“声音往哪走”的问题，两者结合构成完整的音频管理系统。
